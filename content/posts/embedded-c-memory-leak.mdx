組み込みシステムは長期間無人で稼働することが多く、わずかなメモリリークが
数日〜数週間後のシステムクラッシュにつながります。

## 静的アロケーションを優先する

ヒープを使った動的確保（`malloc`/`free`）は便利ですが、断片化やリーク
のリスクを伴います。可能な限り静的配列やメモリプールを使いましょう。

```c
#define MAX_SENSORS 8

typedef struct {
    uint16_t id;
    float value;
} SensorData;

static SensorData sensor_pool[MAX_SENSORS];
```

## 所有権を明確にする

動的確保がどうしても必要な場合は、確保と解放の責任を持つモジュールを
1つに限定するのが鉄則です。

### よくあるアンチパターン

- 確保したポインタを複数モジュールで共有し、誰も `free` しない
- エラーパスで `return` した際に解放処理がスキップされる

```c
void *buf = malloc(size);
if (buf == NULL) {
    return -1; // ここで早期リターンするとリークしない
}

if (process(buf) != 0) {
    free(buf); // エラーパスでも必ず解放する
    return -1;
}

free(buf);
return 0;
```

## 静的解析・実行時ツールの活用

- `cppcheck` や `clang-tidy` で確保/解放の対応漏れを検出
- ターゲットが許せば `valgrind` や ASan でランタイム検証

## まとめ

組み込み開発ではメモリリークの発見が遅れるほど被害が大きくなります。
設計段階での静的アロケーション優先と、所有権の明確化を徹底しましょう。
